# Pantry infra

Ansible playbook + GitHub Actions deploy pipeline that takes a fresh
Ubuntu 22.04 / 24.04 LTS VPS and produces a running Pantry stack:

- Postgres 16 (PGDG) + pg_trgm + pgcrypto, localhost-only, scram auth
- Redis 7 (RDB only, allkeys-lru, localhost-only)
- Node 20 LTS via NodeSource + corepack pnpm 9
- nginx (HTTP/2, gzip, IP-allowlistable admin vhost) + certbot via webroot
- systemd units `pantry-api` (port 4000) and `pantry-admin` (port 4001)
- Encrypted nightly backups (`pg_dump | age | rclone`, 7 daily / 4 weekly /
  3 monthly), with an opt-in restic driver for Hetzner Storage Box
- ufw (22/80/443 only), fail2ban (sshd jail), logrotate, unattended-upgrades

Target host (production): `135.181.15.215` (`api.linhkienkts.com`,
`admin.linhkienkts.com`). The third record `app.linhkienkts.com` is
reserved for the public mobile API surface and is not used yet.

---

## Controller prerequisites

On the operator's laptop:

- Python 3.10+ and `pip install --user ansible ansible-lint`
- `pip install --user 'ansible-pylibssh'` if SSH connection issues arise
- `ansible-galaxy collection install community.general community.postgresql`
- `gh` (GitHub CLI), authenticated
- `rclone` (locally, for testing remotes)
- `age` and `age-keygen` for backup key handling
- `shellcheck` (`brew install shellcheck`)
- `yamllint` (`pip install --user yamllint`)

The operator's SSH key for the VPS lives at `~/.ssh/pantry_vps_ed25519`
(referenced by `inventory.example.ini`). The key has root access; the
deploy user (`pantry`) gets a separate, narrower key (see "Deploy SSH key").

## One-shot provision

```bash
cd infra
cp inventory.example.ini inventory.ini
cp group_vars/all.example.yml group_vars/all.yml
# Edit both: set the host IP, domains, allowlist, age recipient (later),
# rclone remote, deploy_ssh_authorized_key.
ansible-playbook -i inventory.ini site.yml
```

The play is idempotent — re-running on a healthy host should report `changed=0`.
The first run skips the HTTPS server blocks (no certs yet); the second run
renders them now that certbot has obtained certs.

### First-run sequence

1. **`common`** installs base packages, creates the `pantry` user, ufw, fail2ban
2. **`postgres`** installs PG16, generates the `pantry_app` password ONCE and
   persists it to `/etc/pantry/secrets/postgres_password` (root:root, 0600).
   Re-applies read it back — no password rotation on every run.
3. **`redis`** installs Redis 7 with the localhost-only template
4. **`nodejs`** installs Node 20 + corepack pnpm 9
5. **`secrets`** asserts `/etc/pantry/secrets/api.env` and
   `/etc/pantry/secrets/admin.env` exist (operator-managed) and generates
   the age keypair for backup encryption
6. **`app`** installs the systemd units, sudoers fragment, deploy authorized_keys,
   and the nightly backup cron
7. **`nginx`** installs vhosts (HTTP-only on first run, HTTPS once certs are
   present)
8. **`certbot`** obtains TLS certs via `--webroot` and installs the renewal cron

After step 8, re-run the playbook so the nginx role re-renders the HTTPS
server blocks.

## Required secrets

Place these on the host BEFORE running the playbook (the secrets role asserts
they exist):

| Path | Owner | Mode | Source |
|------|-------|------|--------|
| `/etc/pantry/secrets/api.env`     | `pantry:pantry` | 0600 | template: `api/.env.example` |
| `/etc/pantry/secrets/admin.env`   | `pantry:pantry` | 0600 | template: `apps/admin/.env.example` |
| `/etc/pantry/secrets/backup.env`  | `pantry:pantry` | 0600 | optional restic config (see below) |

Auto-generated by the playbook (do NOT pre-place):

| Path | Owner | Mode | Notes |
|------|-------|------|-------|
| `/etc/pantry/secrets/postgres_password` | root:root | 0600 | minted once, re-used forever |
| `/etc/pantry/secrets/age.key` | root:root | 0600 | private key — used by restore.sh |
| `/etc/pantry/secrets/age.pub` | root:root | 0644 | public key — used by backup.sh |

### `api.env` keys (minimum for first deploy)

```
DATABASE_URL=postgresql://pantry_app:<paste from /etc/pantry/secrets/postgres_password>@127.0.0.1:5432/pantry
REDIS_URL=redis://127.0.0.1:6379/0
NODE_ENV=production
PORT=4000
HOST=127.0.0.1
JWT_ACCESS_SECRET=<32+ chars random>
JWT_REFRESH_SECRET=<32+ chars random>
COOKIE_SECRET=<32+ chars random>
PUBLIC_API_BASE_URL=https://api.linhkienkts.com
PUBLIC_ADMIN_BASE_URL=https://admin.linhkienkts.com
# Backup driver fields are read from this file too:
BACKUP_RCLONE_REMOTE=b2:pantry-backups
# BACKUP_AGE_RECIPIENT is read from /etc/pantry/secrets/age.pub by default.
```

### `admin.env` keys (minimum)

```
NODE_ENV=production
PORT=4001
HOSTNAME=127.0.0.1
PANTRY_API_INTERNAL_URL=http://127.0.0.1:4000
NEXT_PUBLIC_API_BASE_URL=https://api.linhkienkts.com
ADMIN_SESSION_COOKIE_SECRET=<32+ chars random>
CSRF_COOKIE_SECRET=<32+ chars random>
```

OAuth (Google / Apple), SMTP, push (FCM/APNs), Sentry, etc. are added once
the underlying accounts exist. Defaults for the first deploy assume none of
those are wired.

## Deploy pipeline

`.github/workflows/deploy.yml` runs on every push to `main`:

1. **test** job — full pnpm test (Postgres + Redis service containers),
   `pnpm -r typecheck`, `prisma migrate deploy` against the test DB
2. **build** job — `pnpm -r build`, packages a tarball with `api/dist/`,
   `apps/admin/.next/standalone/`, `packages/`, and `infra/`
3. **deploy** job — SSH to `pantry@<host>`, extract to
   `/opt/pantry/releases/<sha>/`, run `infra/scripts/deploy-remote.sh`

`deploy-remote.sh` does, in order:

1. `pnpm install --frozen-lockfile` (full deps — Prisma CLI is a devDependency)
2. `pnpm --filter @pantry/api exec prisma migrate deploy`
3. `pnpm prune --prod` ← only AFTER migrate
4. `ln -sfn /opt/pantry/releases/<sha> /opt/pantry/current` (atomic flip)
5. `sudo systemctl restart pantry-api.service pantry-admin.service` (NOT
   `reload` — units are Type=simple with `TimeoutStopSec=30`)
6. Smoke `https://api.linhkienkts.com/health/ready` 5x with backoff; on
   failure flip the symlink back and restart
7. Prune old release dirs (keep last 5)

### Required GitHub repository secrets

| Secret | Example value |
|--------|---------------|
| `DEPLOY_SSH_KEY`  | private key of the deploy user (`pantry@<host>`) — generate locally during first apply |
| `DEPLOY_SSH_USER` | `pantry` |
| `DEPLOY_SSH_HOST` | `135.181.15.215` |
| `API_DOMAIN`      | `api.linhkienkts.com` |

`DEPLOY_SSH_KEY` is NOT the same as `~/.ssh/pantry_vps_ed25519`. That's the
operator's root key. The deploy job uses a separate key with NOPASSWD only
for the six `systemctl` commands listed in `/etc/sudoers.d/pantry`.

### Deploy SSH key (one-time setup)

On the operator's laptop, after `ansible-playbook` succeeds at least once:

```bash
ssh-keygen -t ed25519 -f deploy_pantry_ed25519 -C "github-actions-deploy"
# Append the *.pub to inventory.example.ini's matching group_vars file:
#   group_vars/all.yml → deploy_ssh_authorized_key: ssh-ed25519 ...
ansible-playbook -i inventory.ini site.yml --tags app
# Then add the *private* key contents to GitHub:
gh secret set DEPLOY_SSH_KEY < deploy_pantry_ed25519
gh secret set DEPLOY_SSH_USER --body 'pantry'
gh secret set DEPLOY_SSH_HOST --body '135.181.15.215'
gh secret set API_DOMAIN --body 'api.linhkienkts.com'
# DELETE deploy_pantry_ed25519 from your laptop after the secret is set.
```

## Rollback

Automatic: `deploy-remote.sh` rolls back on smoke-test failure (re-flips the
symlink to the previous release and restarts both units).

Manual:

```bash
ssh pantry@135.181.15.215
cd /opt/pantry/releases
ls -1t                                     # find the desired previous SHA
sudo ln -sfn /opt/pantry/releases/<sha> /opt/pantry/current
sudo /bin/systemctl restart pantry-api.service
sudo /bin/systemctl restart pantry-admin.service
```

## Backups

Default driver: `pg_dump -Fc | age -r <recipient> | rclone copy ...`,
rotated 7 daily / 4 weekly (Sunday) / 3 monthly (1st of month). Cron at
03:17 UTC, installed by the `app` role.

### Manual run / verify

```bash
ssh root@135.181.15.215
/opt/pantry/current/infra/scripts/backup.sh
tail -F /var/log/pantry/backup.log
```

### Restic alternative (Hetzner Storage Box)

If you prefer restic, drop a `/etc/pantry/secrets/backup.env` file with:

```
RESTIC_REPOSITORY=sftp:u123456@u123456.your-storagebox.de:/pantry
RESTIC_PASSWORD_FILE=/etc/pantry/secrets/restic_password
```

Then `apt install restic` once and `backup.sh` will pick it up automatically
on the next cron run. The script `forget --keep-daily 7 --keep-weekly 4
--keep-monthly 3 --prune` after every backup.

### Restore

```bash
# age driver:
sudo /opt/pantry/current/infra/scripts/restore.sh 2026-05-28 daily
# restic driver:
sudo /opt/pantry/current/infra/scripts/restore.sh restic latest
```

The restore script refuses to run non-interactively unless
`RESTORE_NONINTERACTIVE=1` is exported. It reads `DATABASE_URL` from
`/etc/pantry/secrets/api.env`.

## Post-deploy: external uptime monitor

UptimeRobot (or an equivalent off-host pinger) MUST be wired manually after
the first successful deploy. Why off-host: it has to keep working when the
host itself is down.

1. Sign in to https://uptimerobot.com
2. **Add new monitor → HTTP(s)**
3. URL: `https://api.linhkienkts.com/health`
4. Monitoring interval: **5 minutes**
5. Alert contacts: at least one **email** address (free-tier limit: SMS
   requires paid plan)
6. Save and confirm the first probe goes green

The `app` role intentionally does NOT script the UptimeRobot API. The
playbook just prints a reminder during the run.

`/health` (liveness) is used rather than `/health/ready`: that way the
monitor flags hard-down processes even while a dependency (Postgres, Redis)
is mid-restart.

## Locking down the admin vhost

`group_vars/all.yml` ships with `admin_allowlist: []` (TLS-only access for
first deploy). To restrict to specific egress IPs:

```yaml
admin_allowlist:
  - 203.0.113.42/32        # office wifi
  - 198.51.100.0/24        # staff VPN
```

Re-apply with `ansible-playbook -i inventory.ini site.yml --tags nginx`.
The vhost falls back to `deny all` outside the listed CIDRs.

## Pre-apply checklist

Tick each before running `ansible-playbook` for the first time:

- [ ] DNS A records for `api.linhkienkts.com` and `admin.linhkienkts.com`
      point at `135.181.15.215`
- [ ] `~/.ssh/pantry_vps_ed25519` exists locally and the public key is in
      `/root/.ssh/authorized_keys` on the host
- [ ] Ports 22 / 80 / 443 are open at the provider firewall
- [ ] `inventory.ini` and `group_vars/all.yml` are filled in (NOT committed)
- [ ] `letsencrypt_email` is a real address (renewal warnings)
- [ ] `/etc/pantry/secrets/api.env` exists on the host (mode 0600,
      `pantry:pantry`)
- [ ] `/etc/pantry/secrets/admin.env` exists on the host (mode 0600,
      `pantry:pantry`)
- [ ] An rclone remote is configured at
      `/root/.config/rclone/rclone.conf` with the name in
      `BACKUP_RCLONE_REMOTE`
- [ ] (After first apply) UptimeRobot HTTP monitor on
      `https://api.linhkienkts.com/health` at 5-minute interval, email alert

GitHub repo secrets (set after the deploy SSH key is generated):

- [ ] `DEPLOY_SSH_KEY`, `DEPLOY_SSH_USER`, `DEPLOY_SSH_HOST`, `API_DOMAIN`

## Validation

Run all of these from the repo root before committing changes to `infra/`:

```bash
ansible-playbook --syntax-check -i infra/inventory.example.ini infra/site.yml
ansible-lint infra/
yamllint .github/workflows/deploy.yml
bash -n infra/scripts/backup.sh
bash -n infra/scripts/restore.sh
bash -n infra/scripts/deploy-remote.sh
shellcheck infra/scripts/*.sh
```

All seven must exit 0.
